# Anti-patterns
если ты сделал какой то код пользователь пишет о том что так делать не нужно и он сам в коде этого не делает то запиши это сюда по готовому шаблону

```markdown
## ❌ Не Делай {} 
- Причина: 
- Делай: 
```

## ❌ Никогда не используй `process.env` напрямую
- Причина: не работает в Docker, не тестируемо, нет типизации
- Делай: `configService.get('DB_HOST')` через ConfigService

## ❌ Не включай `synchronize: true` в production
- Причина: может дропнуть данные при деплое
- Делай: `synchronize: NODE_ENV === 'development'`

## ❌ Не создавай микросервис «на будущее»
- Причина: излишняя сложность, network overhead, distributed transactions
- Делай: начинай с модульного монолита, выделяй сервис только когда есть явные признаки: отдельная команда, отдельный масштаб, отдельный цикл релизов либо спрашивай у пользователя

## ❌ Не игнорируй идемпотентность в Kafka consumers
- Причина: события могут прийти дважды (rebalance, retry)
- Делай: проверяй `message_id` или бизнес-ключ перед обработкой

## ❌ Не обрабатывай Kafka событие без проверки идемпотентности
- **Почему**: ребалансировка/ретраи могут доставить событие дважды
- **Делай**: проверяй `message_id` или бизнес-ключ перед обработкой

## ❌ Не используй одинаковые порты для test и production
- **Причина**: конфликт при запуске docker compose
- **Делай**: используй ports 6400+ для test DB (6433, 6434, ...)

## ❌ Не полагайся на init SQL автоматически  
- **Почему**: docker-entrypoint-initdb.d выполняется только при ПЕРВОМ запуске контейнера
- **Симптом**: "column does not exist" после перезапуска
- **Делай**: проверяй схему вручную, добавляй колонки если нужно

## ❌ Не игнорируй gRPC readiness vs actual methods
- **Почему**: waitForReady() может вернуть true, но методы возвращают Internal Error
- **Причина**: DB schema неполная
- **Делай**: тестируй реальные методы, не только readiness
