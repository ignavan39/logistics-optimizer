## Strict Mode Activation

Если пользователь пишет:
- «сделай фичу Х»
- «реализуй интеграцию с Y»
- «оптимизируй сервис Z»

→ Агент **обязан** сначала:
1. Задать уточняющие вопросы (если не хватает контекста)
2. Предложить план с декомпозицией (по шаблону выше)
3. Дождаться подтверждения плана
4. Только потом писать код

**Запрещено**: сразу генерировать код без плана для фич >50 строк.

# Golden Rule: Tests First

**Перед написанием любой бизнес-логики:**
1. Создай unit-тест для чистой функции/сервиса
2. Создай интеграционный тест для Kafka/gRPC/DB взаимодействия
3. Создай e2e-сценарий для сквозного потока (если затрагивает >1 сервиса)

## Документация

Поддерживать актуальность документации в `docs/`:

| Файл | Обновлять при |
|------|--------------|
| `docs/SERVICES.md` | Добавлении/удалении сервиса, изменении ответственности |
| `docs/COMMUNICATION.md` | Изменении gRPC методов, добавлении Kafka событий |
| `docs/DATABASE.md` | Изменении схемы БД (новые таблицы, колонки, FK) |
| `docs/API.md` | Добавлении REST endpoints |
| `docs/FEATURES.md` | Добавлении новых фич/паттернов |

Проверка после изменений:
```bash
pnpm build && pnpm typecheck
```
